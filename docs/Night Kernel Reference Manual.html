Section 0 - About the Night Kernel

A synopsis of Night:

The Night kernel has been designed from the ground up to be a full 32-bit drop-in replacement for the original 16-bit kernel of the FreeDOS operating system, and is intended to be loaded by the existing FreeDOS bootloader. Night is built with handcrafted assembly code using the NASM assembler, employs linear memory addressing, and operates in Protected Mode on the Intel x86 architecture. The completed kernel will allow users to enjoy modern abilities such as task switching between applications, protected memory, and increased overall performance in a DOS environment while maintaining full compatibility with their existing DOS applications.

Currently, the Night Kernel is a work in progress; most of the code written to this point is providing internal functionality, and as such the kernel is not yet suitable for everyday use. Night will not, at present, run user applications.



Days before the Night:

Although the Night project itself began in June of 2015, its roots go back much further. Its creator originally got the idea to build a custom operating system from scratch after the lackluster release of Microsoft's Windows 95. Initially, this idea manifest as a simple shell with the goal of bringing multitasking and other modern features to older hardware... and "older hardware" is no joke - the original project featured character graphics and was coded in IBM BASICA on a PC XT!

After a subsequent upgrade to a 80486 machine, the project - then known simply as "PCOS" - evolved as well. The next version was the first to feature color graphics (albeit in only 16 shades), a rudimentary UI and windowing toolkit, and mouse control. It was initially coded with QBASIC but quickly migrated to Visual BASIC as it outgrew the former language. As the project continued, it naturally became desirable to add true preemptive multitasking. Alas, the problem with that is that running multiple simultaneous DOS programs at once, reliably, under a mere shell would be much too complex to be feasible.

After a series of code refinements which pressed the limits of the current development rig, , the main development rig was upgraded once again, this time to a 667 MHz Celeron machine. The project's source code was migrated to PowerBASIC, and the added processing power of the new PC made possible the incorporation of a virtual machine of sorts which would allow applications compiled for it to run concurrently. The result was a tech demo featuring multiple applications coded in an Assembly language designed especially for the virtual machine, all running under a preemptive scheduler. In time, the project - which had by now taken on the name Exelyx - grew to have not only its own assembler, but a BASIC-to-assembly-bytecode converter as well.

Exelyx was evolved to one last form before being retired permanently in August of 2016; a DPMI-powered MS-DOS program with a code base rewritten in Watcom C. Though Exelyx is dead, some key sections of code developed during its lifespan have now been adapted, in part, for use in a new and refined platform; one which will rely on no underlying operating system or drivers, but shall supply the facilities to manage every aspect of the user's machine completely on its own.

That project is Night.

Section 1 - Building and Running the Kernel

Before we begin, keep in mind these directions are written based on a small variety of Linux-based workstations used by the Night developers. This is not to say, however, that this is the only way to build the kernel; you can adapt these directions to whatever other compatible environment you like. The sole applications needed are an assembler, NASM, and a small utility called Xenops (included in the Night repo) which makes it easy to update build numbers automatically. If you'd rather not use Xenops, Night can be built without its convenience, but you will need to modify the makefile. There are no pre-made libraries used in the creation of the Night Kernel; the repo contains all the code necessary to build the kernel from scratch without the need to download additional modules.



There are multiple ways to take Night for a test drive:

Booting the pre-made VirtualBox image
In the builds folder of the repo, you will find a VirtualBox disk image named Night.vdi; a 16 MiB disk with a base-only install of FreeDOS 1.2 (sans sources) and the default kernel.sys swapped out for the Night Kernel. This disk image is fixed-size to allow for easier mounting under the host operating system and is ready to boot a VirtualBox virtual machine.

To create a virtual machine suitable for testing out Night, the default DOS configuration offered by VirtualBox can, for the most part, be used without alteration, however you should name the machine Night for compatibility with the included makefile.



Booting the pre-assembled kernel binary
The repo's builds folder is also home to a pre-assembled kernel.sys, which you can copy over the existing FreeDOS kernel on any bootable FreeDOS medium, then just boot from that bootable media as you normally would.



Building from source
If you're running in a Linux environment and have both VirtualBox and NASM installed, you can use the included makefile found in the root of the repo to streamline building and testing. The rules it accepts are as follows:

Table 1.1 - Make options
Rule Result
buildnight Assembles the kernel using NASM. After assembling, you will have a file called kernel.sys in the builds directory which can be copied over top of the existing FreeDOS kernel on any bootable FreeDOS disk.
updateimage Mounts the virtual disk image night.vdi and copies kernel.sys to it. You will be prompted for your root password as a normal function of this option.
run Runs the kernel in a VirtualBox virtual machine.
all Performs all of the above steps: builds Night, mounts the included disk image and copies to it the newly build kernel, then launches the virtual machine.






Section 2 - The Kernel Debug Menu

After booting into the kernel, the Kernel Debug Menu will appear along with a few other tasks designed to demonstrate the kernel's multitasking abilities. In the completed product, this menu will only be accessible after being enabled via ConfigBits, but as the kernel in its current form does not yet run applications, it makes sense to have this menu display by default.

The menu provides the following options:

Table 2.1 - Debug Menu selection summary
Menu Selection Description
System Info Provides details about the system, including the kernel version, CPU description, and the addresses of various data lists the kernel maintains.
PCI Devices Details the PCI devices detected on the system and allows browsing of the PCI register information for each using the Page Up and Page Down keys.
Memory Details Provides a dump of the memory list used to track both free and allocated physical RAM.
RAM Browser An interactive memory browser. Use the Plus and Minus keys on the number pad to scroll through memory addresses or use Page Up or Page down to jump by larger chunks. Typing a hexadecimal address and pressing enter will jump instantly to that memory location. Keep in mind that some modern systems have protected ranges of RAM used by UEFI or SMM/TPM software which are not permitted to be read. Accessing these with the RAM Browser can cause a triple-fault and reset the system.
Task Browser Allows browsing of all slots in the Task List. Use Page Up and Page Down to change the task number, the Plus and Minus keys on the number pad to adjust the priority of running tasks, Spacebar to toggle a task's suspended state, or Enter to kill the task entirely. Keep in mind there is nothing preventing the Debug Menu itself from being suspended or killed, leaving no way to interact with the kernel until the system is restarted.
Reboot Reboots the machine.

When the kernel is built, NASM will generate a map file, kernel.map, containing the addresses of all statically allocated structures and variables in the kernel. Any memory blocks allocated at runtime will not show here, but pointers to them can be found under the System Info option in the Debug Menu or by directly accessing the pointers in the tSystem struct.

The makefile will start VirtualBox with the --debug-command-line option, which enables the debugger console of the virtual machine. Section 4 - Coding Conventions Used in Night

"C ... all the power of assembly language programming with all the ease of assembly language programming."
  - Stuart Cheshire, Tricks of the Mac Game Programming Gurus

Assembly has gotten a bad rap as a language both difficult to use and ill-suited for many modern programming projects. Perhaps this is due to Assembly code of days gone by being written in long, rambling blocks with little structure of any kind, or maybe certain assemblers of old being rather rigid in the syntax they accepted, but there is no reason why this must be. With a little structure and finesse, Assembly can be forged with no worse readibility than comparable C code.

To illustrate this, let's take a look at a function from the Night source code and discuss its structure. Below, we have an example from the Memory Manager:


  section .text
  MemAllocateAligned:
     ; Returns the address of a memory block of the requested size aligned to the value specified, or zero if unavailble
     ;
     ; input:
     ;   Requesting task number
     ;   Requested memory size in bytes
     ;   Alignment necessary
     ;
     ; output:
     ;   EAX - Address of requested block, or zero if call fails
     ;   EDX - Result code

Every function in Night begins by stating it is part of the .text section of the final binary image, so that the assembler knows to group this function in with the rest of the runnable code. Following this designation is the function label, which in this case is MemAllocateAligned. Immediately after, there is a short description of what the function does, and what it expects for input. Since the return results of all Night funtions are passed back to the caller in registers, the registers used by the function and what information they will contain can be found here as well.




     push ebp
     mov ebp, esp

 Next EBP, the current base pointer, is preserved to the stack and reloaded with the current stack pointer. This provides a reference so that we can use local variables in the function and it also provides a handy way to reference the various parameters passed from the caller. This is the standard entry procedure for every Night function which accepts input parameters or uses internal variables.




     ; allocate local variables
     sub esp, 20
     %define blockAddress                 dword [ebp - 4]
     %define blockAddressAligned          dword [ebp - 8]
     %define blockLeadingSize             dword [ebp - 12]
     %define blockTrailingSize            dword [ebp - 16]
     %define blockRequestedSize           dword [ebp - 20]

 Now that EBP has been preserved and set, offsets from it may be defined to represent any local variables needed in this function. To do this, we first must adjust the stack pointer to clear out enough spack in the stack area for the variables we will need: in this case 20 bytes, enough space for five double-word values. After that, the individual variables can be defined. Note that defining the variables with labels is not technically required (using bare references like dword [ebp - 4] in the code directly is perfectly valid) but using the % define directive essentially tells NASM something to the effect of, "Every time you see blockTrailingSize in the code below, go ahead and substitute it with dword [ebp - 16]" which makes the code much easier for us humans to read.





     ; make sure alignment is 2 or greater to avoid both wasted time (1) and division by zero errors (0)
     cmp dword [ebp + 16], 2
     jb .Exit

     ...

 Next in this function is an example of how passed parameters are accessed. Values pushed onto the stack by the caller begin at [EBP + 8] and ascend from there. Since all of Night's functions expect 32-bit parameters, this means that the first parameter passed will live at [EBP + 8], the second at [EBP + 12], the third at [EBP + 16], and so on. In the example above, the cmp command referencing [EBP + 16] is comparing the third parameter (or alignment necessary, as specified further above in the function's input and output description) to the value 2.

A handy way of remembering which EBP references are which is locals are lower than EBP, and parameters are positive.




     ...

     ; set the return values and exit
     mov edx, 0
     mov eax, blockAddress

 And now, skipping ahead to the end, the work of this function is complete. Remember those registers mentioned above in the function descrption? This is where they get set prior to exiting the function.




     .Exit:
     mov esp, ebp
     pop ebp
  ret 12

 And finally, it's time to exit! The mov and pop lines comprise the standard exit procedure for nearly all functions in the Night kernel. In short, they undo the entry procedure performed at the start of this function and restore the stack to the state it was prior.

The ret 12 instruction will not only return to the caller, but also takes an additional twelve bytes off the stack. Why twelve bytes? The answer lies in the number of input parameters passed to this function. MemAllocateAligned takes three double-word parameters as input, and we can clear out the space they use (in essence popping them all back off the stack in one fell swoop) by specifying the total size at the end of the ret instruction. By the same token, if a function instead takes no parameters, a simple ret could be used since there would be no parameter space on the stack of which to be rid.

Section 5 - The Night Kernel Driver Model

Driver Structure

Every Night driver begins with a driver header, laid out as follows:

Table 5.1 - Driver header format
Offset Description
0x00 signature: 'N', 0x01, 'g', 0x09, 'h', 0x09, 't', 0x05, 'D', 0x02, 'r', 0x00, 'v', 0x01, 'r', 0x05
0x10 driverFlags

The Night Kernel uses a modular approach to drivers, allowing individual driver files to be added or removed at will without having to alter any other sections of the kernel's code. The signature field allows the kernel to discover the driver regardless of where it may lie in the kernel's driver address space.

The driverFlags field is a bank of flags to identify attributes about this driver. Currently, the following bit definitions are supported:

Table 5.2 - Driver flags bit definitions
Bit Description
00 Supports character device interface
01 Supports block device interface
02 - 28 Reserved
29 PCI driver
30 Filesystem driver
31 Legacy driver

Immediately after this structure can be found additional data depending on the type of driver.



Driver Types

Night drivers are divided into three main categories:

Filesystem Driver
A filesystem driver is, as its name implies, a driver to add kernel-level support for a particular file system.

Fields specific to this driver type:
Table 5.3 - File system driver sub-header format
Offset Description
0x14 Filesystem Type

Filesystem Type refers to the single-byte code found in the partition table which identifies the kind of filesystem held in the volume. For example, a FAT 12 volume would have 0x01 here, and a FAT 32 volume would show 0x0B. An exhaustive list of these codes may be found at this external site.



Legacy Driver
A legacy driver is intended for a device with no attachment to an established detection interface, such as older ISA or other onboard devices such as the PS/2 controller. For example, the kernel may not directly support the ISA bus, but devces attached to it may still be detected by testing the response received from certain I/O ports associated with the devices. Every legacy driver found by the kernel will have its init code run, unlike PCI drivers which only have their init code executed if a device is found which matches their capabilities.

Due to the nature of Legacy Drivers, they contain no specially structured data after the driver header.



PCI Driver
Night will probe the PCI bus and run the init code of the driver only if its capabilities (determined by the fields in the table below) are appropriate for the device in question.

Fields specific to this driver type:
Table 5.4 - PCI driver sub-header format
Offset Description
0x14 classMatch
0x18 subclassMatch
0x1C progIfMatch
0x20 WriteCodePointer
0x24 ReadCodePointer

The classMatch, subclassMatch, and progIfMatch fields are all used to match the driver to the device found on the PCI bus. A value of 0xFFFF in any of these fields means that the driver matches all values of that field; all other values are taken literally. For example, for the ATA driver, classMatch is set to 1, subClassMatch is set to 1, and ProgIfMatch is set to 0xFFFF, meaning that this driver can handle any devices with Class 1, Subclass 1, and any ProgIf (Program Interface) value at all.

The function pointers WriteCodePointer and ReadCodePointer are how the kernel calls the read and write functions of the driver.

Section 7 - The List Manager

Managing smooth operation of a computer system necessitates the kernel keep track of numerous points of data. To ease that task, a library of code was created to manage arrays of stored data: the List Manager. Browsing the kernel source illustrates just how many items in which the List Manager has a hand in tracking; everything from drives and partitions to the PCI devices contained in the computer. A list, in this context, is simply a series of blocks of data of a constant size (called elements) stored in series, preceeded by a header, which contains a handful of 32-bit values which describe characteristics of the list.

Lists are structured as follows:

Table 7.1 - List Manager header format
Offset Name Description
0x00 signature A string, "list", identifying the beginning of a list structure
0x04 elementSize Size in bytes of each element in this list
0x08 elementCount Total number of elements in this list
0x0C listSize Total size of this list (header and data space) in bytes
0x10 List data






Section 8 - Memory Management in Night

Memory allocation is an important part of any operating system, and Night is no different. Upon boot, Night will probe the BIOS memory map using function 0xE820 of interrupt 0x15. After obtaining these results, the single largest block of RAM found in the map will be selected for use as the entire physical memory pool. This memory block holds the memory list at its lower end, which is used by the List Manager to track what sections of physical RAM have been allocated to which processes. The format of an individual memory list entry is shown below.

Table 8.1 - Format of a Memory List element
Offset Name Description
0x00 address The starting physical address of this block of memory
0x04 size The size of this physical block
0x08 task The task number which allocated this block. A zero in this field means the block is free for use, ergo memory requests specifying zero as the allocating task will be rejected.

Although most versions of MS-DOS support First-Fit, Best-Fit, and Last-Fit allocation strategies, Night currently uses Best-Fit exclusively. Memory requests are fulfilled from the top-down, meaning that the first block allocated will be located at the end of the memory area and subsequent allocations will grow downwards toward the Memory List, much like stack operation.

Another difference is the time at which free memory blocks are combined; MS-DOS performs this task upon every allocation attempt, but Night combines upon every deallocation.






Section 9 - Multitasking, Night Style

One of the main benefits Night brings to the DOS experience is multitasking. Some older operating systems common to the DOS era would use a technique called Cooperative Multitasking, which relied on programs to yield their control of the CPU back to the operating system before another program could get time to do its job. This is not the way Night does things.

Preemptive Multitasking is a system by which task switches are initiated by an outside source, usually a hardware interrupt provided to the CPU, which means applications cannot monopolize the processor. This is the technique used by Night.



An Overview of Tasking on Night

The Programmable Interval Timer is set to fire 256 times per second, and its interrupt handler simply adds one to the tSystem.ticksSinceBoot value, then checks if the global multitasking switch, tSystem.taskingEnable, is non-zero. If it is, task-switching logic is performed to determine the next valid task to be run in a round-robin manner.

In computing, as in life, certain tasks are sometimes more crucial to complete than others. Extending this principle to the digital workspace, Night's task management structure contains a priority value which dictates how many turns at the CPU the task will have in a row before being switched out for the next waiting task. Currently the priority levels supported range from 0 (one turn before switch) to 255 (an over-the-top value which is mainly there simply for future-proofing. One caveat exists: since the PIT interrupts fire 256 times per second, maxing out the priority value to 255 can cause near-total lockup of other running tasks. Therefore, it is not recommended to use a priority value over 3 so that other tasks do not visibly suffer from lost CPU time. A future version will incorporate a global execution cap, preventing tasks from being set to too high of a priority.

When a task is switched in, Night reads the CPU's Time Stamp Counter and stores the result in switchInLow and switchInHigh. When it is switched back out, the TSC is read once again and the original switch-in values are subtracted from this reading. The resulting value is stored in cycleCountLow and cycleCountHigh and can be used as an estimated number of cycles the task spent doing work during its turn at the CPU. This information can be used by Task Manager-type programs to easily calculate how well tasks are utilizing the timeslices they are allocated.

Although the x86 architecture does provide a method of performing multitasking in hardware, Night - like many other operating systems - does not use it. Instead, the faster software method is used via a single Task State Segment which is updated at every task switch.

Night tracks task data using a list of tTaskInfo structures, each of which is 96 bytes in length.



Table 9.1 - The tTaskInfo structure
Offset Name Size Description
0x00 pageDirAddress dword The page directory address for this task. This is copied to CR3 upon task switch.
0x04 entryPoint dword The initial starting value of EIP for this task.
0x08 esp dword The user-level stack pointer.
0x0C esp0 dword The kernel stack pointer.
0x10 stackAddress dword The memory address of the user stack as returned by MemAllocate().
0x14 kernelStackAddress dword The memory address of the kernel stack as returned by MemAllocate().
0x18 priority byte The priority value for this task. May be 0 - 255, but values over 3 can cause noticible lag in other tasks.
0x19 turnsRemaining byte A temporary value used by the task switching code to track how many turns at the CPU this task has left before a task switch will occur. The value of priority is copied into this location initially and then subsequently decremented each pass through the task switching code, at which point priority is copied into this value again for the next time this task is swapped in.
0x1A unused word Free for future allocation
0x1C switchInLow dword The low DWord of the TSC value when this task is switched to.
0x20 switchInHigh dword The high DWord of the TSC value when this task is switched to.
0x24 cycleCountLow dword The low DWord of the total cycles this task used during its most recent turn at the CPU.
0x28 cycleCountHigh dword The high DWord of the total cycles this task used during its most recent turn at the CPU
0x2C spawnedBy dword The task number of the task which spawned this task.
